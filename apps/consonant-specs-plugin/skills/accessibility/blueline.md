# Blueline Fill — Accessibility Annotation Generator

Fill AI-assisted sections of a blueline accessibility annotation in Figma.

## When to Use

- User says "fill the blueline" or "fill the blueline for {frame name}"
- Or consonant-mcp `get_blueline_request` returns a pending request

## Prerequisites

- Figma Desktop Bridge must be connected (figma_execute available)
- A blueline scaffolding must already exist on the canvas (generated by the Consonant Tools plugin)
- The scaffolding contains frames with "Awaiting AI fill..." text nodes

## Execution Flow

### Phase 1 — Gather Context

1. Take a screenshot of the target design frame with `figma_take_screenshot`
2. Read the node tree via `figma_execute` (depth 4-5) to get names, types, sizes, hierarchy
3. Extract all TEXT node content for heading/name analysis
4. Locate the "Accessibility Annotations" sidebar frame on the canvas
5. Find all sections containing "Awaiting AI fill..." text

### Phase 2 — Analyze and Generate

For each section that needs filling, analyze the design and generate content.

Reference docs for accuracy:
- `skills/accessibility/wcag-patterns.md` — ARIA roles and keyboard patterns
- `skills/accessibility/carousel-a11y.md` — carousel-specific rules
- `skills/accessibility/landmarks-guide.md` — landmark mapping

#### Heading Hierarchy
- Analyze all TEXT nodes by fontSize, fontWeight, and visual position
- Determine H1-H6 structure considering: logo as potential H1, eyebrow vs heading distinction
- Output: heading level tag + first ~50 chars of text content

#### Landmarks
- Map top-level visual sections to ARIA landmark roles
- Common mappings: navigation → nav, header area → banner, footer → contentinfo, main content → main, named sections → region
- Include aria-label and aria-roledescription where needed

#### Accessible Names
- For each focusable element, provide a context-aware accessible name
- Go beyond visible text: "Learn more" → "Learn more about {context}"
- Provide reusable patterns, not just one-off values

#### Alt-Text
- Identify IMAGE fills on rectangles/frames
- Use `figma_take_screenshot` on individual image nodes to see content
- Write descriptive alt-text or mark as decorative

#### ARIA Roles & Attributes
- Identify UI component patterns (tablist, carousel, dialog, menu, etc.)
- Assign roles, aria-* attributes, and state management following WAI-ARIA APG
- Include tabindex management for composite widgets

#### Keyboard Patterns
- For each identified component pattern, specify keyboard behavior
- Follow WAI-ARIA Authoring Practices Guide patterns exactly
- Include: arrow keys, Home/End, Tab, Enter/Space, Escape as applicable

#### DOM Strategy
- Only generate if dynamic components detected (carousel, tabs, accordion, etc.)
- Recommend DOM structure: all panels present vs lazy load, inert/aria-hidden usage

#### Auto-Rotation
- Only generate if carousel/slideshow detected
- Specify: prefers-reduced-motion handling, interval, pause triggers (focus, hover), resume rules, manual stop behavior

#### Screen Reader Notes
- Only generate if platform-specific behavior differs from standard ARIA
- Cover: VoiceOver (iOS/macOS), TalkBack (Android), Narrator (Windows)

### Phase 3 — Write to Canvas

For each section, use `figma_execute` to:
1. Find the placeholder text node containing "Awaiting AI fill..."
2. Replace its content with the generated text
3. Add numbered entries where applicable
4. The purple left border and AI markers are already in the scaffolding — don't modify them

For detail cards (H1 Strategy, DOM Strategy, Auto-Rotation):
1. Find the detail card frame by name
2. Replace the placeholder text with generated content

### Phase 4 — Complete

1. Call `blueline_complete` on consonant-mcp if it was triggered via MCP
2. Report to user what was filled and how many sections

## Quality Rules

- Every filled section is AI-generated — the purple markers already signal this
- Reference WAI-ARIA APG for all ARIA and keyboard recommendations
- Use `figma_take_screenshot` for visual context — don't guess from node names alone
- Conditional generation: skip Auto-Rotation if no carousel, skip Screen Reader if no platform diffs
- Provide patterns, not just values: "Learn more about + {eyebrow}" not just "Learn more about Acrobat"
