# Blueline Fill — Accessibility Annotation Generator

Fill AI-assisted sections of a blueline accessibility annotation in Figma.

## When to Use

- User says "fill the blueline" or "fill the blueline for {frame name}"
- Or consonant-mcp `get_blueline_request` returns a pending request

## Prerequisites

- Figma Desktop Bridge must be connected (figma_execute available)
- A blueline scaffolding must already exist on the canvas (generated by the Consonant Tools plugin)
- The scaffolding contains frames with "Awaiting AI fill..." text nodes

## Architecture — Scaffold + JSON + Render

The blueline fill uses a three-stage pipeline for speed and consistency:

1. **Gather** — one `figma_get_blueline_data` call returns structural scan + screenshot + knowledge (Phase 1)
2. **Analyze** — 10 parallel agents each produce structured JSON for their cards (Phase 2)
3. **Render** — one `figma_render_blueline` call stamps all cards with guaranteed consistent styling (Phase 3)

Agents return pure JSON — they never call Figma tools. This makes them fast, parallelizable, and style-consistent.

## Execution Flow

### Phase 1 — Gather Structure (One Call)

Call `figma_get_blueline_data`. It returns structural data + orchestration instructions:
- `structuralScan` — textNodes, repeatingGroups, imageNodes, iconFrames, pairedStacks, overlays, focusableElements
- `focusOrder` — existing focus order entries
- `bluelineCards` — card IDs and names to fill
- `instructions.agentGroups` — agent group definitions with scanFields and knowledgeKeys
- `instructions.scanSummary` — counts per scan field (for conditional skipping)
- `instructions.skipHint` — which agent groups can be skipped (empty scan data)
- `availableKnowledge` — list of knowledge keys available via `figma_get_knowledge`
- Attached screenshot image (no separate `figma_take_screenshot` call needed)

**Knowledge is NOT in this response** — it's too large. Fetch it per agent group in Phase 2.

### Phase 2 — Fetch Knowledge + Parallel Agent Dispatch

**Step 1:** For each non-skipped agent group, call `figma_get_knowledge({ keys: group.knowledgeKeys })` to get the knowledge content. You can call these in parallel.

**Step 2:** Dispatch agents in parallel using the Agent tool. **All a11y agents MUST use `model: "sonnet"`** — this saves significant cost while maintaining accuracy since agents follow structured knowledge files. The orchestrator (main conversation) stays on whatever model the user is running.

Use `instructions.agentGroups` from the response — each group defines:
- `cardKeys` — which cards this agent fills (1-4 cards per agent)
- `scanFields` — which structural scan fields to include (trim the rest)
- `knowledgeKeys` — which `knowledge[key]` entries to embed in the prompt

**Trimming:** For each agent, build a trimmed scan object containing only the fields listed in `scanFields`. This keeps agent prompts focused and fast.

**Conditional skipping:** Check `instructions.skipHint` — it lists agent groups whose scan data is entirely empty (e.g., no carousels → Motion & Media skippable). For skipped groups, do NOT spawn an agent. Instead, directly produce `items: []` with a note for each cardKey. This saves significant time on simple pages. The `instructions.scanSummary` shows exact counts per scan field.

**Group filtering:** `agentGroups` is pre-filtered to only include groups whose cards exist in this blueline. If the user only selected 5 categories, you may see fewer than 10 groups.

**Each agent MUST return JSON as an array of card results:**
```json
[
  {
    "cardKey": "landmarks",
    "items": [
      { "title": "banner — Site Header", "desc": "Contains logo, primary nav, and search" },
      { "title": "main — Hero Content", "desc": "Primary content area with CTA" }
    ],
    "notes": ["WCAG 1.3.1 — programmatic landmark roles identify page regions"],
    "warnings": []
  },
  {
    "cardKey": "skipNav",
    "items": [
      { "title": "Skip to main content", "desc": "Skip link needed — 8 interactive elements before main content" }
    ],
    "notes": ["WCAG 2.4.1 — mechanism to bypass repeated blocks"],
    "warnings": []
  }
]
```

#### Agent Prompt Template

```
You are an accessibility specialist analyzing a Figma design for blueline annotations.

## Your Cards: {CARD_KEYS_LIST}

## Context (screenshot attached above)
{TRIMMED_STRUCTURAL_SCAN_JSON — only the scanFields for this agent}

## Domain Knowledge
{FULL CONTENT of knowledge[key] for every key in knowledgeKeys — concatenated}

## Analysis Instructions
{CARD-SPECIFIC_INSTRUCTIONS — from the section below, for each cardKey}

## Output Format
Return ONLY a valid JSON array. One object per cardKey:
[
  {
    "cardKey": "{CARD_KEY}",
    "items": [{ "title": "...", "desc": "..." }, ...],
    "notes": ["WCAG X.X.X — ..."],
    "warnings": ["..."]
  }
]

Rules:
- titles: short, specific element/concept names (e.g. "H1 — Buy now with Creative Cloud")
- desc: one sentence explaining the recommendation or observation
- notes: cite specific WCAG SC codes with plain-language summary
- warnings: only for actual issues found in the design — not hypothetical risks
- If a card category doesn't apply, return items: [] with a note explaining why
- Return one object per cardKey — do not skip any
```

#### Card-Specific Analysis Instructions

Include these instructions in each agent's prompt alongside the domain knowledge:

**Heading Hierarchy agent:**
- If scan data exists: check `textNodes` list (sorted by font size descending). Largest text = H1 candidate. Use `fontFamily` to distinguish headings (Adobe Clean Display = heading) from body text (Adobe Clean Regular = not heading). Use `depth` for nesting.
- Apply rules: one H1 per page, never skip levels, logo is NOT H1, eyebrow is NOT a heading
- Card titles in repeating groups are usually one level below the section heading above them
- Output items as indented heading tree: title = "H{level} — {first ~50 chars of text}", desc = role/purpose of that heading

**Landmarks agent:**
- Map top-level visual sections to ARIA landmark roles
- Include `aria-label` when multiple landmarks of the same type exist
- Don't over-landmark: not every frame needs a role
- Title = landmark role + label, desc = what section it maps to

**Accessible Names agent:**
- If scan data exists: check `iconFrames` where `hasTextChild: false` — icon-only buttons needing `aria-label`
- Check `focusableElements` cross-referenced with `textNodes` — elements near vague text ("Learn more", "See all") need contextual names
- For vague links: harvest context from nearest heading in `textNodes`
- Provide reusable patterns, not one-off values: "Learn more about + {card title}"
- Title = element description, desc = recommended accessible name strategy

**Alt-Text agent:**
- If scan data exists: check `imageNodes` — `isFullBleed: true` = decorative (`alt=""`), `isFullBleed: false` = content (needs alt), `hasTextSibling: true` = hero with overlay (background decorative)
- Functional images (icons in buttons): describe the function, not the appearance
- Title = image element name/location, desc = recommended alt text or strategy

**ARIA Roles agent:**
- If scan data exists: check `repeatingGroups` to identify component patterns (tabs, pagination, card grid), `overlays` for modals, `pairedStacks` for accordions
- Look up exact roles, states, and properties from matching pattern in the knowledge file
- Include tabindex management strategy
- Title = component/element name, desc = ARIA contract summary

**Keyboard Patterns agent:**
- Use same pattern identification as ARIA Roles, then look up keyboard contract
- List every key and what it does — don't summarize, be specific
- Include focus model (roving tabindex, focus trap, standard)
- Title = component/interaction name, desc = key bindings and behavior

**DOM Strategy agent:**
- Only generate items if dynamic components detected (modals, carousels, accordions, tabs)
- Specify: panels in DOM vs lazy-loaded, show/hide method, inert/aria-hidden usage
- If no dynamic components: return `items: []` with note "No dynamic components detected"
- Title = component name, desc = DOM rendering strategy

**Auto-Rotation agent:**
- Only generate items if carousel/slideshow detected in scan data or screenshot
- Specify: prefers-reduced-motion handling, interval, pause triggers (focus, hover), resume rules
- If no carousel: return `items: []` with note "No auto-rotating content detected"
- Title = rule/behavior name, desc = implementation requirement

**Color Contrast agent:**
- Check text fills against background colors — flag anything below 4.5:1 (normal) or 3:1 (large)
- Check non-text UI elements (borders, icons) against background — flag below 3:1
- Note: actual ratio computation should be done with tools — flag the concern and cite WCAG SC
- Title = element or color pair, desc = ratio finding and verdict

**Forms agent:**
- Only generate items if forms/inputs detected in the design
- Check for visible labels, required indicators, error state designs, autocomplete needs
- If no forms: return `items: []` with note "No form elements detected"
- Title = form element or pattern, desc = requirement and recommendation

**Target Size agent:**
- Measure interactive elements — flag anything below 24x24px
- Check spacing between adjacent interactive elements
- Pay special attention to icon-only buttons, pagination, close buttons
- Title = element with measured size, desc = finding and fix

**Reflow & Text Spacing agent:**
- Check if responsive/mobile variants exist in the design
- Flag fixed-width containers, side-by-side layouts without stacking breakpoint
- Flag fixed-height text containers that could clip on text spacing override
- If single fixed-width design: return items noting reflow requirements
- Title = layout area, desc = reflow requirement

**Language agent:**
- Identify primary language of design content
- Flag any text in a different language
- Check for language selector components
- Title = language concern, desc = requirement

**Time-Based Media agent:**
- Only generate items if video/audio elements detected
- Check for caption controls, transcript links, audio description tracks
- If no media: return `items: []` with note "No time-based media detected"
- Title = media element, desc = requirement

**Skip Navigation agent:**
- Check how many interactive elements appear before main content
- If navigation has 3+ links before content, skip link is needed
- Title = skip link type, desc = implementation note

**Page Title agent:**
- Determine appropriate page title from design content/heading
- Check for SPA-style navigation that needs title updates
- Title = page/view name, desc = recommended title format

**Reduced Motion agent:**
- Identify animations/transitions in the design (carousel, parallax, hover effects)
- Classify each as essential or decorative
- If no animations detected: return `items: []` with note "No animations detected"
- Title = animation/effect name, desc = classification and reduced-motion alternative

**Consistent Navigation agent:**
- Only meaningful when multiple page designs exist in the file
- Compare navigation order and labels across pages
- If single page design: return items noting consistency requirements for the nav
- Title = navigation element, desc = consistency finding

**Screen Reader Notes agent:**
- Only generate items if design contains complex patterns (modals, carousels, custom widgets)
- Standard ARIA patterns (buttons, links, headings) work consistently — no platform note needed
- Cover VoiceOver (Name→Role→State), TalkBack (Role→Name→State), Narrator (heading nav via H key)
- Title = platform or behavior name, desc = platform-specific note

### Phase 3 — Render

After all 10 agents return, flatten their JSON arrays into one `cards` object and call `figma_render_blueline`:

```js
// Each agent returns an array of card results — flatten into one map
const cards = {};
for (const agentResult of agentResults) {
  const cardArray = JSON.parse(agentResult); // array of { cardKey, items, notes, warnings }
  for (const card of cardArray) {
    cards[card.cardKey] = {
      items: card.items,
      notes: card.notes || [],
      warnings: card.warnings || []
    };
  }
}

// One call renders everything with consistent styling
figma_render_blueline({ cards });
```

Cards with `items: []` are skipped (carousel, DOM strategy, screen reader when not applicable).

### Phase 4 — Focus Order (Sequential)

Focus order cannot be parallelized — it requires sequential traversal of the design:

1. Analyze the screenshot and node tree to determine the correct tab order
2. For each interactive element, assign an accessible name
3. Call `figma_create_focus_annotations` with the ordered list

This runs AFTER Phase 3 since it needs a stable canvas.

### Phase 5 — Complete

1. Call `blueline_complete` on consonant-mcp if it was triggered via MCP
2. Report to user what was filled and how many sections

## Quality Rules

- Every filled section is AI-generated — the purple markers already signal this
- Agents use the preloaded knowledge from `figma_get_blueline_data` — not general knowledge
- Screenshot is included in the `figma_get_blueline_data` response — no separate call needed
- Conditional generation: skip Auto-Rotation if no carousel, skip Screen Reader if no platform diffs
- Provide patterns, not just values: "Learn more about + {eyebrow}" not just "Learn more about Acrobat"
- Each agent returns pure JSON — never calls Figma tools directly
- One `figma_render_blueline` call ensures all cards have identical visual styling
